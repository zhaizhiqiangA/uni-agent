核心修改内容：

1. **容器 CMD 适配**（_FixedCmdDockerEnvironment）
   - 根因：minisweagent 的 DockerEnvironment._start_container 硬编码 CMD 为 ["sleep", timeout]
   - sweb 镜像 ENTRYPOINT 为 /bin/bash，拼接后执行 /bin/bash sleep 2h，bash 把 /usr/bin/sleep ELF 当脚本读，报 "cannot execute binary file"（exit 126），容器立即退出
   - R2E-Gym 镜像 ENTRYPOINT 为 nvidia_entrypoint.sh，直接 exec CMD，原始 sleep 2h 可正常执行
   - 修复：启动容器前通过 docker inspect 检测镜像 ENTRYPOINT，以 bash 结尾时用 ["-lc", "sleep <timeout>"] 作为 CMD，否则用原始 ["sleep", "<timeout>"]
   - 验证结果：sweb 镜像（astropy）容器正常存活 137 步；R2E-Gym 镜像容器正常，score=1.0 resolved=True

2. **parallel_infer.py 异常容错**
   - 根因：generate_sequences 所有 rollout 失败时抛 RuntimeError("All rollouts failed")，parallel_infer.py 未捕获导致进程崩溃
   - 修复：try/except 包裹 asyncio.run(_generate())，捕获 RuntimeError 后输出 warning 日志和 Resolved 0/N 统计

3. **step_limit 调试日志**
   - 添加 SWE_AGENT_MAX_TURNS 环境变量读取和 agent.config.step_limit 值的日志输出
   - 验证确认环境变量正确传递到 DefaultAgent（日志显示 step_limit=100）

4. **遗留问题**（未修复，记录备查）
   - Qwen3.5 processor.apply_chat_template 不支持 OpenAI 格式的 tool_calls 消息
   - 当 gateway 全量编码带 tool_calls 的多轮消息时触发 TypeError，fallback 后导致 "System message must be at the beginning"
   - 触发条件：_is_request_context_prefix 匹配失败时走 _encode_full 全量编码路径
   - 影响：agent 迭代轮次过多时（>75 轮）可能触发此问题
