核心修改内容（从上一次正式存档 archive_20260524_0620 至今，含临时断点 checkpoint_20260524_0935 合并汇总）：

1. **tool_calls arguments 格式归一化**
   - 问题：mini-swe-agent 通过 litellm 发送的 assistant 消息中，`tool_calls[].function.arguments`
     是 JSON 字符串（符合 OpenAI API 规范），但 Qwen 的 chat template 使用 `|items` 迭代要求 dict 对象
   - 根因：gateway 的 `_normalize_message` 未对 arguments 做类型转换，直接透传给模板引擎
   - 影响：导致 TypeError "Can only get item pairs from a mapping"，随后触发 `_apply_chat_template`
     wrapper 的 dummy_user 回退，产生二次错误 "System message must be at the beginning"；
     同时前缀匹配失败，每轮都走全量编码路径
   - 修复：新增 `_normalize_tool_call_arguments` 函数，在 `_normalize_message` 中将 arguments 从
     string 解析为 dict；解析失败时保持原样（兼容其他模板）
   - 效果：TemplateError 从 60 次降为 0，前缀匹配恢复正常，后续轮次走增量编码

2. **vLLM 后端异常 HTTP 状态码映射**
   - 问题：`_backend.generate()` 抛出的 ValueError（如上下文长度超限）未捕获，FastAPI 统一返回 500
   - 影响：litellm/tenacity 看到后 500 后盲目重试（最多 10 次，约 4-5 分钟），导致 session 卡住
   - 修复：捕获 `_backend.generate()` 所有异常，ValueError → 400（invalid_request_error），
     其他 → 500（internal_error），返回 OpenAI 格式错误响应
   - 效果：客户端能区分可恢复/不可恢复错误；但 mini_swe 的 tenacity 重试机制
     （非 litellm 的 max_retries）仍会重试 400，需后续处理

3. **上下文溢出场景验证（RESPONSE_LENGTH=4096）**
   - 使用 mini_swe runner + Qwen3.5-9B + SWE-bench 2 样本，故意设小 response_length
   - 约 20 轮后对话历史超出 max_model_len (9216 tokens)
   - 当前行为：gateway 返回 400，tenacity 重试 10 次后放弃，runner 以 reward=0 结束 session，
     framework 标记为 failed（数据未写入 TQ）
   - 遗留：tenacity 重试浪费 4-5 分钟，可通过 MSWEA_MODEL_RETRY_STOP_AFTER_ATTEMPT=1 禁用

4. **SWE-rebench 推理验证（临时断点记录，代码已回退）**
